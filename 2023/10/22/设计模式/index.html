设计模式

发表于2023-10-22更新于2023-10-22
阅读量: 深圳

设计模式类别

单例模式 (Singleton Pattern):

用途：

  • 资源管理：
    • 用途： 单例模式可以用于管理共享的资源，如数据库连接池、线程池等，以确保在整个应用程序中只有一个资源管理实例。
    • 案例： 数据库连接池。在一个高并发的应用中，使用单例模式管理数据库连接池可以有效地减少连接创建和销毁的开销，提高性能。
  • 配置管理：
    • 用途： 单例模式可以用于管理应用程序的配置信息，确保配置信息在整个应用程序中是一致的。
    • 案例： 日志管理器。一个应用可能需要记录日志，并且日志的配置信息（如日志级别、输出路径等）在整个应用中应该是一致的。通过使用单例模式，可以确保只有一个日志管理器实例，避免配置信息的不一致性。
  • 线程池管理：
    • 用途： 在多线程环境中，单例模式可以用于创建和管理线程池，确保线程的控制和资源的合理利用。
    • 案例： Java 中的 Executors 类。Executors 提供了一系列工厂方法用于创建各种类型的线程池，而这些线程池的实例是通过单例模式管理的。
  • 日志记录：
    • 用途： 单例模式可以用于记录应用程序的日志信息，确保日志信息的一致性和集中管理。
    • 案例： Log4j 中的 Logger 类。Logger 类是 Log4j 中负责记录日志的核心类，通过单例模式确保只有一个 Logger 实例，以便在整个应用程序中统一管理日志记录。
  • 缓存管理：
    • 用途： 单例模式可以用于管理应用程序中的缓存，确保缓存的一致性和有效性。
    • 案例： Spring 框架中的 CacheManager。CacheManager 负责管理应用中的缓存，通过单例模式确保只有一个 CacheManager 实例，以便对缓存进行统一管理。

1. 懒汉式单例模式

懒汉式是指在需要时才创建实例，如果实例已经存在，就直接返回。这样可以延迟对象的创建，但需要考虑多线程安全问题。

2. 饿汉式单例模式

饿汉式是指在类加载时就创建实例，因此不存在多线程安全问题。但可能会造成资源浪费，因为实例在整个生命周期中都被创建，即使未被使用。

工厂模式 (Factory Pattern):

简单工厂模式、工厂方法模式和抽象工厂模式。下面分别给出这三种方式的代码案例：

1. 简单工厂模式

简单工厂模式通过一个工厂类，根据传入的参数来创建不同类型的对象。这种方式的优点是简单易用，但缺点是不够灵活，新增产品需要修改工厂类。

2. 工厂方法模式

工厂方法模式定义一个用于创建对象的接口，由子类决定实例化的类。这样每个产品都有一个对应的工厂类，更符合开闭原则。

3. 抽象工厂模式

抽象工厂模式提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们的具体类。适用于产品族的场景。

这三种工厂模式分别适用于不同的场景，选择合适的工厂模式取决于项目的需求和架构。

建造者模式 (Builder Pattern):

建造者模式是一种创建型设计模式，它的目的是将一个复杂对象的构建过程与其表示分离，使得同样的构建过程可以创建不同的表示。建造者模式包括以下主要角色：产品（Product）、抽象建造者（Builder）、具体建造者（ConcreteBuilder）和指挥者（Director）。

下面通过代码案例来解释建造者模式：

1. 产品类

产品类是要构建的复杂对象，它包含了多个部件。

2. 抽象建造者

抽象建造者定义了构建产品的抽象接口，包括构建不同部件的方法。

3. 具体建造者

具体建造者实现了抽象建造者接口，负责实际构建产品的各个部件。

4. 指挥者

指挥者负责调用建造者的方法，按照一定的构建顺序构建产品。

5. 使用建造者模式

使用建造者模式时，客户端通过指挥者调用建造者的方法，构建产品。

这个例子中，通过建造者模式，客户端可以构建一个复杂对象（Product），并且可以灵活地改变构建的过程和表示，而不需要直接调用产品的构造方法。建造者模式适用于构建对象结构复杂、部件间关系多样的情况。

原型模式 (Prototype Pattern):

原型模式是一种创建型设计模式，其主要目的是通过复制现有对象来创建新的对象，而不是通过构造函数创建。原型模式的关键是通过复制（克隆）一个现有对象来创建新对象，从而避免了耗时的初始化过程。

下面通过代码案例来解释原型模式：

2. 具体原型类

3. 客户端使用原型模式

客户端通过调用原型的克隆方法来创建新对象。

在这个例子中，通过原型模式，客户端通过克隆现有对象来创建新对象，而不是通过构造函数。这使得对象的创建更加灵活，并且可以避免一些对象初始化的开销。需要注意的是，Java 中的 clone() 方法是浅拷贝，如果对象中包含引用类型的成员变量，需要注意其克隆的方式。如果需要深拷贝，可以考虑实现深拷贝逻辑或使用序列化方式进行深拷贝。

适配器模式 (Adapter Pattern):

适配器模式是一种结构型设计模式，它允许接口不兼容的类能够一起工作。适配器模式主要包括三种角色：目标接口（Target）、被适配者（Adaptee）、适配器（Adapter）。

下面通过代码案例来解释适配器模式：

1. 目标接口

目标接口是客户端期望使用的接口，它定义了客户端代码调用的方法。

2. 被适配者

被适配者是已经存在的类，它的接口与目标接口不兼容。

3. 适配器

适配器是连接目标接口和被适配者的类，它实现了目标接口，并包含了一个被适配者的实例。

4. 客户端使用适配器模式

客户端通过目标接口调用适配器的方法，实际上是在调用被适配者的方法。

在这个例子中，Adaptee 类的 specificRequest 方法是客户端不能直接调用的，因为它的接口与目标接口 Target 不兼容。通过适配器 Adapter，客户端可以通过调用 Target 接口的 request 方法来实际调用 Adaptee 的 specificRequest 方法。

适配器模式在系统中可以起到“桥梁”作用，将不同接口的类协同工作，使得原本不兼容的类能够协同工作。这在系统升级、组件复用等场景中非常有用。

装饰者模式 (Decorator Pattern):

装饰者模式是一种结构型设计模式，它允许向一个现有的对象添加新的功能，同时又不改变其结构。装饰者模式通过创建装饰类，将装饰类嵌套在被装饰者类中，以透明的方式扩展对象的功能。

下面通过代码案例来解释装饰者模式：

1. 抽象组件

抽象组件是被装饰者和装饰者的共同接口，它定义了被装饰者和装饰者都必须实现的方法。

2. 具体组件

具体组件是实现了抽象组件接口的具体类，它是被装饰者的基础。

3. 抽象装饰者

抽象装饰者是实现了抽象组件接口的抽象类，它包含一个对抽象组件的引用，并通过构造函数传入被装饰者。

4. 具体装饰者

具体装饰者是实现了抽象装饰者的具体类，它通过扩展抽象装饰者来添加新的功能。

5. 客户端使用装饰者模式

客户端可以通过组合不同的装饰者来扩展被装饰者的功能。

在这个例子中，ConcreteComponent 是被装饰者，ConcreteDecoratorA 和 ConcreteDecoratorB 是具体装饰者。客户端可以通过组合不同的装饰者来实现不同的功能组合。装饰者模式在不改变原有对象结构的情况下，动态地扩展其功能，使得代码更加灵活和可维护。

代理模式 (Proxy Pattern):

代理模式是一种结构型设计模式，它允许通过代理对象控制对另一个对象的访问。代理模式通常用于控制对对象的访问，以实现一些额外的操作，例如延迟加载、权限控制、监控等。

下面通过代码案例来解释代理模式：

1. 主题接口

主题接口是被代理对象和代理对象都必须实现的接口，它定义了一些操作。

2. 具体主题

具体主题是实现了主题接口的具体类，是被代理对象。

3. 代理类

代理类实现了主题接口，并包含一个对具体主题的引用。代理类可以在执行操作前后进行一些额外的处理。

4. 客户端使用代理模式

客户端通过代理对象来访问具体主题。

在这个例子中，RealSubject 是具体主题，Proxy 是代理类。客户端通过代理对象 Proxy 访问具体主题 RealSubject，并且在执行操作前后进行了一些额外的处理。

代理模式在许多场景中都有应用，例如远程代理、虚拟代理、保护代理等。通过代理，可以实现一些横切关注点的处理，使得被代理对象的功能更加灵活和可扩展。

观察者模式 (Observer Pattern):

观察者模式是一种行为型设计模式，它定义了一种一对多的依赖关系，使得当一个对象的状态发生变化时，所有依赖于它的对象都能够得到通知并自动更新。观察者模式通常被用于实现分布式事件处理系统。

下面通过代码案例来解释观察者模式：

1. 主题接口

主题接口定义了被观察者（主题）和观察者之间的通信接口。

2. 具体主题

具体主题是实现了主题接口的具体类，它维护了一个观察者列表，并在状态发生变化时通知所有观察者。

3. 观察者接口

观察者接口定义了观察者的通知方法。

4. 具体观察者

具体观察者是实现了观察者接口的具体类，它实现了在主题状态发生变化时的更新逻辑。

5. 客户端使用观察者模式

客户端创建具体主题和具体观察者，并将观察者注册到主题中。当主题状态发生变化时，所有注册的观察者都会收到通知。

在这个例子中，ConcreteSubject 是具体主题，ConcreteObserver 是具体观察者。当主题状态发生变化时，通过调用 notifyObservers 方法通知所有注册的观察者，观察者收到通知后执行相应的更新逻辑。

观察者模式提供了一种松耦合的设计，使得主题和观察者可以独立演化而不相互影响。这样的设计在实现事件处理、消息传递等场景中非常有用。

策略模式 (Strategy Pattern):

策略模式是一种行为型设计模式，它定义了一系列算法，将每个算法封装起来，并使它们可以互相替换。策略模式使得算法的变化独立于使用算法的客户端。

下面通过代码案例来解释策略模式：

1. 策略接口

策略接口定义了一组算法的通用接口。

2. 具体策略

具体策略是实现了策略接口的具体类，每个具体策略实现了一种算法。

3. 上下文

上下文类包含一个对策略接口的引用，并可以在运行时切换不同的具体策略。

4. 客户端使用策略模式

客户端可以创建不同的具体策略，并通过上下文对象设置不同的策略。

在这个例子中，ConcreteStrategyAdd、ConcreteStrategySubtract 和 ConcreteStrategyMultiply 分别是具体的策略，客户端可以通过创建不同的具体策略并设置给上下文来实现不同的算法。

策略模式允许算法的变化独立于使用算法的客户端，使得系统更加灵活且易于扩展。这种模式在需要在运行时选择算法的场景中非常有用，例如排序算法、图形处理等。
